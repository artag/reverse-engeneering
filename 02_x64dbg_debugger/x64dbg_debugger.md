### 4.2. Желательно. Сделать снимок на виртуальной машине

В любой современной программе для виртуализации есть функция снимков состояния
(Snapshots). Это встроенный механизм, который работает как "точка восстановления"
в компьютерной игре. Если вы сделаете снимок, запустите вирус, и он полностью уничтожит
систему, вы сможете вернуть виртуалку к исходному состоянию за один клик.

Снимки в разных программах

- VirtualBox: Виртуалка в списке -> Кнопка Вкладки  -> Snapshots -> Take
- VMware Workstation: VM (верхний угол) -> Snapshot -> Take Snapshot
- Hyper-V: ПКМ по виртуал6ке -> Checkpoint

Порядок работы со снимками

1. Настроить чистую систему
2. Установить все инструменты (Detect It Easy, x64dbg)
3. Добавить папки в исключения
4. Сделать снимок

Как восстановить

1. Выключить виртуалку
2. Меню снимков -> Восстановить (Restore / Revert) к чистому снимку.

**Важное правило безопасности** перед запуском файла **отключить в виртуалке**:

1. Общие папки (Shared Folders)
2. Общий буфер обмена (Shared Clipboard)
3. Drag and Drop (перетаскивание файлов мышкой)
4. Сетевой адаптер (переведите в режим «Host-only» или отключите совсем)

### 4.3. Утилита `DIE` (Detect It Easy)

Узнать информацию о ломаемом файле. Пример, что можно узнать

- Type: `PE` - Portable Executable - тип файла, запускаемый файл
- ImageBase - адрес, откуда программа начинает выполяться
- EntryPoint - Адрес точки входа (смещение от ImageBase)
- Size
- Compiler - компилятор
- Linker - линкер
- Может быть еще указан packer - упаковщик, может затруднить reverse

Кнопка `PE` и прочие кнопки раскрывает более подробную информацию по файлу.

От версии Detect It Easy зависит объем потенциально доступной информации по файлу.

Главное тут выяснить какой битности файл 32 или 64.
В зависимости от битности запускается x32dbg или x64dbg.

### x32dbg/x64dbg. Про TLS Callbacks

TLS Callbacks (обратные вызовы Thread Local Storage) — это специальные функции в
ОС Windows, которые выполняются до передачи управления на главную точку входа программы.

Изначально этот механизм был создан Microsoft для инициализации локальных данных
потоков (Thread Local Storage) при запуске приложения или создании нового потока.
Однако в сфере реверс-инжиниринга и анализа вредоносного ПО (Malware Analysis) TLS
Callbacks часто используются разработчиками защит и вирусописателями как мощный метод
антиотладки и сокрытия кода.

Если вредоносный или защищенный код упакован с использованием TLS Callbacks:

- Загрузчик Windows (NTDLL) находит в PE-заголовке таблицу TLS.
- Код внутри TLS Callback выполняется скрытно в фоновом режиме до остановки отладчика на Entry Point.
- Этот скрытый код успевает проверить наличие отладчика в системе, завершить процесс, повредить память или расшифровать вредоносную полезную нагрузку.
- Отладчик при этом даже не успевает показать первую инструкцию программы.

xdbg при включенном TLS Callback может корректно обрабатывать и останавливать
выполнение на таких функциях.

Опцию остановки на TLS Callbacks часто советуют **выключать** в повседневной работе по двум главным причинам:

- потеря времени на легитимных программах
- ложные срабатывания компиляторов

#### Мусорные остановки в обычном софте

1. Современные компиляторы MSVC/Rust/Go генерируют TLS Callbacks автоматически.
Они нужны для безопасной инициализации стандартных библиотек и потоков
Этот код абсолютно легитимен, безопасен и не содержит логики программы.
При включенной опции отладчик будет "застревать" в системном коде библиотек
при каждом перезапуске программы.

#### Сбивает фокус с анализа

Подавляющее большинство программ (около 95%), которые вы будете анализировать, не являются вредоносными или защищенными протекторами.
Постоянная остановка на TLS заставляет тратить время на чтение скучного
инициализационного кода компилятора, отвлекая от реальной точки входа (Entry Point).

#### Когда рекомендуется включить обратно опцию TLS Callback

Включайте эту опцию строго вручную и только в следующих сценариях:

- Анализ малвари (Malware Analysis): Если вы исследуете вирус, криптор или лоадер.
- Программы под протекторами: Если исследуемый файл упакован защитой (VMProtect,
Themida, UPX, ASPack). Они часто прячут там распаковщики и проверки на отладчик.
- Странное поведение: Программа закрывается сразу после загрузки в отладчик,
а до Entry Point вы так и не дошли.

#### Правило хорошего тона в x32dbg/x64dbg

Держать опцию TLS Callbacks **выключенной**, а при подозрении на трюки антиотладки -
переходить на вкладку «Символы» (Symbols) и проверять наличие адресов TLS визуально,
либо временно активировать галочку.

### x32dbg/x64dbg. Про Ignored Exceptions

Вкладка Ignored Exceptions - это список кодов исключений (ошибок), которые отладчик
будет пропускать мимо себя, отдавая их на обработку самой программе.

Если исключение находится в этом списке, x32dbg не будет вставать на паузу и прерывать выполнение. Программа обработает его сама через свои внутренние механизмы (SEH/VEH).

#### Зачем нужен список Ignored Exceptions

Многие программы и протекторы используют исключения как часть своей штатной логики.
Если не добавить их в игнорируемые, вы столкнетесь со следующими проблемами:

- Постоянные раздражающие остановки: Программа генерирует сотни мелких ошибок в секунду.
Отладчик будет виснуть на каждой, и придется бесконечно нажимать F9.

- Трюки антиотладки (Anti-Debugging): Защита намеренно вызывает ошибку (например,
STATUS_BREAKPOINT или 0xC0000005), чтобы проверить реакцию.
Если отладчик перехватит её и не передаст программе, защита поймет, что её анализируют,
и завершит процесс.

#### Что будет если добавить `0x00000000 - 0xFFFFFFFF`

Если добавить условную маску `0x00000000 - 0xFFFFFFFF` в список игнорируемых.

- Плюс: Отладчик вообще перестанет отвлекать вас на системные сбои, внутренние проверки
защит, сетевые ошибки и логи компилятора. Программа будет выполняться на максимальной
скорости. Это полезно для взломов.

- Огромный минус: Вы полностью ослепнете. Если программа совершит критическую ошибку
(например, обратится к несуществующей памяти и упадет), вы не узнаете, на какой
конкретно инструкции это произошло.
Отладчик просто молча закроется вместе с процессом, не создав точку останова на месте
крушения.

Режим отсутствия игнорирования исключений полезен для режима работы debug.

### Использование `DIE`

Открываем ломаемый файл в `DIE` (Detect It Easy).

Смтрим на:

- ImageBase - адрес, откуда программа начинает выполяться
- EntryPoint - Адрес точки входа (смещение от ImageBase)

Нам нужно начать анализ в x32dbg с адреса равным:

```txt
ImageBase = 0x00400000
EntryPoint = 0x000014e0

Искомый адрес = 0x00400000 + 0x000014e0 = 0x004014e0
```

### Описание x32dbg/x64dbg

Основное окно:

- 0 колонка: Graphical view of jumps
- 1 колонка: RAM Memory Addresses
- 2 колонка: OPCODE - machine code в hex представлении
- 3 колонка: Assembly code (human readable code)
- 4 колонка: комментарии (ручные и созданные xdbg)

Программа x32dbg/x64dbg является disassembler, выполняет convert
машинного кода (Opcode) в assembly code (human readable code).

1 и 3 колонка основные, с чем производится работа.

`JE`, `JLE`, `JNZ`, `JMP`, ... - все являются инструкциями jump.

`JMP` - unconditional jump, все остальные - conditionals

### Использование x32dbg/x64dbg

При загрузке программы обычно ее выполнение начинается с Operating System Code, а не
с кода crackme.

crackme начинается с адреса `0x004014e0` (выяснили в `DIE`).

При нажатии на `Run` (F9) программа начинает выполнение и происходит останов на
адресе `0x004014e0` (EntryPoint программы).

Повторное нажатие F9 запустит выполнение программы до ее завершения.

`Restart` перезапускает выполнение программы с Operating System Code.

F8 или `Step Over` - выполняет по одной инструкции.
